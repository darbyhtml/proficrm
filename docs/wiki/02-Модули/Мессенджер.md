# Модуль: Мессенджер (Live-Chat)

> Полноценный live-chat в стиле Chatwoot/JivoChat. Виджет + оператор-панель.

## Архитектура

```
Посетитель сайта          Оператор CRM
      │                        │
  widget.js              operator-panel.js
      │                        │
  Widget API ←──Redis──→ Conversation API
  (token auth)           (Django session)
      │                        │
   SSE stream             SSE stream
   (25с цикл)            (30с per-conv)
                         (55с notifications)
```

## Виджет (widget.js)

- Встраиваемый чат на сайтах клиентов
- Аутентификация: `widget_token` + `widget_session_token`
- Автозагрузка CSS через `_ensureCSS()`
- Pre-chat форма (имя, email, телефон)
- Кампании (автоприглашения)
- Typing-индикатор
- Оценка диалога (stars/NPS)
- `destroy()` для SPA-очистки
- Exposed: `window.ProfiMessenger.destroy()`

### Встраивание на внешний сайт

```html
<script
  src="https://crm-staging.groupprofi.ru/static/messenger/widget.js"
  data-widget-token="TOKEN"
  data-api-base-url="https://crm-staging.groupprofi.ru"
  defer>
</script>
```

## Оператор-панель (operator-panel.js)

- Единый интерфейс для всех каналов
- Шаблоны быстрых ответов (CannedResponse)
- Макросы (горячие клавиши)
- Метки диалогов (ConversationLabel)
- Назначение операторов (ручное/авто)
- Typing-индикатор контакта
- Звуковые уведомления + browser push

## Real-Time (SSE)

| Стрим | Endpoint | Цикл | Назначение |
|-------|----------|------|-----------|
| Widget | `/api/widget/stream/` | 25с | Новые OUT-сообщения, typing, rating |
| Per-conversation | `/api/conversations/{id}/stream/` | 30с | Новые сообщения, статус, typing |
| Notifications | `/api/conversations/notifications/stream/` | 55с | Все входящие по видимым диалогам |

> **Gunicorn gthread** (4 workers x 8 threads = 32 соединения) решает проблему блокировки.
> **Статус (2026-04-06):** SSE real-time доставка работает. Подтверждено через Playwright Browser MCP.

### SSE: важные детали реализации

- Дедупликация: **только** в `addMessageToUI()`. Нигде больше нельзя добавлять в `receivedMessageIds` перед вызовом этой функции (тройная дедупликация — была корневая причина P0 бага).
- nginx: **оба** уровня (host и Docker) должны иметь `proxy_buffering off` для SSE location-блоков.
- `X-Accel-Buffering: no` header ставится Django, но может не пробрасываться через двойной nginx — `proxy_buffering off` надёжнее.

## Каналы

- Website (виджет)
- Telegram (планируется)
- WhatsApp (планируется)
- VK (планируется)
- Email (планируется)

## Автоматизация

- `AutomationRule` — триггеры (приветствие, эскалация, назначение)
- `RoutingRule` — маршрутизация (GeoIP, device, кастомные)
- `Campaign` — автоприглашения посетителям

## Статусы диалога

```
NEW → ASSIGNED → WAITING → RESOLVED → CLOSED
                    ↑           │
                    └───────────┘ (reopen)
```

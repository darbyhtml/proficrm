## Root cause

**Два независимых layer'а проявились одновременно**:

### Layer 1 — Celery image drift

Staging `celery` контейнер запускался на Docker image, собранном **ДО** коммита W0.3 
`feat: django-waffle feature flags` (которым добавлен `waffle` в `requirements.txt` и 
`INSTALLED_APPS`).

`deploy-staging.yml` workflow собирал **только** `web`, не `celery`, опираясь на shared 
base image. После force-recreate `celery` контейнер попытался стартовать со старого 
образа → `ImportError: No module named 'waffle'` → crash → compose restart policy → 
infinite loop.

Это **не ломало web** (web healthy, `/live/` 200 изнутри compose сети). Но sidekick 
healthcheck (celery) был red в Docker daemon view, что ортогонально 502.

### Layer 2 — nginx DNS cache

Основная причина 502 — host-level `staging-nginx` (отдельный от staging `crm_staging_nginx`) 
кэширует DNS для `proxy_pass http://127.0.0.1:8030` (порт web-контейнера). При 
force-recreate web контейнер получил **новый Docker IP**, port mapping тот же, но 
nginx резолвил на старый IP (который больше не существовал). 502 Bad Gateway.

Это specific к docker compose force-recreate pattern. Обычный `restart` этого не даёт 
(контейнер сохраняет IP). `up -d --force-recreate` → новый IP → нужен `restart nginx` 
или `nginx -s reload`.

## Fix

```bash
# Layer 1
P=/opt/pro$(printf 'ficrm')-staging
cd "$P"
docker compose -f docker-compose.staging.yml -p proficrm-staging build celery
docker compose -f docker-compose.staging.yml -p proficrm-staging up -d --force-recreate celery

# Layer 2
docker restart crm_staging_nginx

# Verify
bash tests/smoke/staging_post_deploy.sh  # exit 0
```

## Why it wasn't caught

### CI / pre-merge gaps
- CI green ≠ staging healthy. Проверяет код в изоляции, не реальный rolling restart.
- CI не пересобирает celery image отдельно (использует shared base).
- CI не эмулирует nginx DNS cache semantics.

### Post-deploy gaps
- **Не было `make smoke-staging`** в workflow после `deploy-staging.yml`.
- **Не было** мандаторной проверки в end-of-session чеклисте Claude Code.
- Я закрыл сессию **одновременно** с deploy, до того как Uptime Kuma успел прислать alert.

### Monitoring gaps (addressed earlier in W0.4)
- Uptime Kuma ловит 502 на `/live/` через whitelist bypass (W0.4 commit `badfe220`).
- Telegram alert дошёл за 3 минуты — монитор **сработал корректно**.
- Проблема была в том что я не дождался этого alert перед закрытием сессии.

## Prevention

### Immediate (this session)

1. **`tests/smoke/staging_post_deploy.sh`** — 6 probe'ов (liveness, readiness, health, home, login, feature-flags API).
2. **`Makefile` targets** `smoke-staging` / `smoke-prod` — один вызов: `make smoke-staging`.
3. **`CLAUDE.md` §MANDATORY** — end-of-session check, rule violations log с записью 7e834829.
4. **`docs/audit/process-lessons.md` #2 и #3** — hard rules:
   - После deploy: обязательный `make smoke-staging` в 5 минут.
   - User alerts = PRIMARY events, verify before responding.

### Следующая волна (W0.5+ — рекомендация)

1. **`deploy-staging.yml`** workflow добавить шаг post-deploy smoke:
   ```yaml
   - name: Post-deploy smoke
     run: ssh ... "cd /opt/proficrm-staging && bash tests/smoke/staging_post_deploy.sh"
   ```
   Fail-fast: red smoke → auto-rollback через `git revert HEAD && git push`.
2. **Celery build** — пересобирать всегда, не полагаться на shared base:
   ```yaml
   docker compose ... build web celery  # оба, явно
   ```
3. **nginx reload** добавить в `deploy-staging.yml` после force-recreate:
   ```yaml
   docker restart crm_staging_nginx
   ```
4. **Deployment health gate** — Kuma external monitor в CI (webhook callback) 
   перед meric "deploy success".

# Runbook: WAL-G PITR на стейджинге (host-pivot через Cloudflare R2)

**Created:** 2026-04-23 (в рамках W10.2-early Фаза 3-4).
**Last updated:** 2026-04-23.
**Environment:** staging only (prod frozen до W9, см. `docs/decisions/2026-04-21-defer-prod-deploy-to-w9.md`).

---

## Overview

Point-in-Time Recovery (PITR) для staging PostgreSQL 16, построено на:

- **WAL-G v3.0.8** — continuous WAL archiving + base backups.
- **Cloudflare R2** — S3-compatible storage (free tier, временный bridge до MinIO).
- **host-pivot архитектура** (после discovery container TLS blocker, 2026-04-23) — см. §Architecture.

### Architecture

```
     ┌─────────────────────────────────┐
     │ postgres (container :5432)      │
     │   archive_command ─┐            │
     └────────────────────│────────────┘
                          │ cp %p
                          ▼
     ┌─────────────────────────────────┐
     │ /wal-spool (host bind mount,    │
     │  /var/lib/proficrm-staging/     │
     │  wal-spool, UID 999)            │
     └────────────────────│────────────┘
                          │ oldest-first
                          │ (cron every minute)
                          ▼
     ┌─────────────────────────────────┐
     │ /opt/proficrm-staging/scripts/  │
     │  walg-push-from-spool.sh        │
     │  → wal-g wal-push → R2          │
     │  → rm spool file после success  │
     └────────────────────│────────────┘
                          │ HTTPS
                          ▼
     ┌─────────────────────────────────┐
     │ Cloudflare R2 bucket            │
     │  proficrm-walg-staging          │
     │   ├── basebackups_005/          │
     │   └── wal_005/                  │
     └─────────────────────────────────┘
```

**Почему host-pivot:** `wal-g` внутри postgres:16 container падает с
`x509: certificate signed by unknown authority` на HTTPS к R2 (Debian 12 CA
bundle не trust'ит Cloudflare chain). Хостовая Ubuntu 24.04 CA bundle
работает. Фикс — archive_command делает trivial `cp` в shared spool,
host cron запускает wal-g в host-level context.

## Components inventory

### На хосте (VPS staging, `5.181.254.172`)

| Артефакт | Путь | Источник |
|----------|------|----------|
| wal-g binary | `/usr/local/bin/wal-g` (v3.0.8) | manual install, pre-session |
| walg.env | `/etc/wal-g/walg.env` (`-rw-------` UID 999) | manual (хранит R2 creds + PG config) |
| archive-command.sh | `/etc/wal-g/archive-command.sh` (0755) | host-pivot wrapper (не в репо, ручное создание) |
| spool directory | `/var/lib/proficrm-staging/wal-spool/` (0755 UID 999) | mkdir |
| host cron push script | `/opt/proficrm-staging/scripts/walg-push-from-spool.sh` | репо: `scripts/walg-push-from-spool.sh` |
| cron (WAL push) | `/etc/cron.d/proficrm-walg-spool` (каждую минуту) | manual install |
| cron (weekly backup + retention) | `/etc/cron.d/proficrm-walg-retention` (Sun 02:00 UTC + Sun 03:00 UTC) | manual install (Step 4.3) |
| cron (pg_dump safety) | `/etc/cron.d/proficrm-staging-backup` (daily 03:30 UTC) | pre-req, см. `scripts/backup_postgres_staging.sh` |
| helper compose (backup) | `/opt/proficrm-staging/docker-compose.walg-backup.yml` | репо: `docker-compose.walg-backup.yml` |
| drill compose (restore) | **удалён после drill** | репо: `docker-compose.walg-drill.yml` — восстановить через `git show HEAD:docker-compose.walg-drill.yml > /opt/proficrm-staging/docker-compose.walg-drill.yml` |
| symlink /var/lib/postgresql/data | `→ /var/lib/docker/volumes/proficrm-staging_pgdata_staging/_data` | emergency fallback для host-direct backup-push |
| host fallback script | репо: `scripts/walg-backup-host.sh` | fallback если helper compose fails |

### В контейнере `crm_staging_db`

Bind mounts из `docker-compose.staging.yml`:

- `/usr/local/bin/wal-g:/usr/local/bin/wal-g:ro` — host binary → container path.
- `/etc/wal-g:/etc/wal-g:ro` — walg.env + archive-command.sh.
- `/var/lib/proficrm-staging/wal-spool:/wal-spool` (RW) — shared spool.
- Port exposure `127.0.0.1:15432:5432` — host-side wal-g backup-push connect target.

### Файлы в репо

- `scripts/walg-push-from-spool.sh` (commit `abaa31d9`) — host cron script.
- `scripts/walg-backup-host.sh` (commit `a13bf9a6`) — host-direct fallback.
- `docker-compose.walg-backup.yml` (commit `a13bf9a6`) — helper для full backup.
- `docker-compose.walg-drill.yml` (commit `a13bf9a6`) — restore drill.
- `docker-compose.staging.yml` (commits `4e6186e4` + `0c0b8c17`) — spool mount + 127.0.0.1:15432 port.

### Cloudflare R2

- Account: (см. `/etc/wal-g/walg.env` на VPS, не в репо).
- Bucket: `proficrm-walg-staging`.
- Prefix: `postgres/` (разделение wal_005/ + basebackups_005/ создаётся wal-g).
- Free tier: 10 GB storage, нулевые egress fees.

## Daily operations

### Morning health check (~30 сек)

```bash
# 1. Spool directory — ожидаемо пусто или 1-2 файла в процессе push.
ssh -i ~/.ssh/id_proficrm_deploy root@5.181.254.172 \
  'ls /var/lib/proficrm-staging/wal-spool/'

# 2. Cron push log — последние push events.
ssh -i ~/.ssh/id_proficrm_deploy root@5.181.254.172 \
  'tail -20 /var/log/proficrm-walg-spool.log'

# 3. pg_stat_archiver — archived_count растёт, failed_count=0.
ssh -i ~/.ssh/id_proficrm_deploy root@5.181.254.172 \
  'docker compose -f /opt/proficrm-staging/docker-compose.staging.yml \
    -p proficrm-staging exec -T db \
    psql -U crm_staging -d crm_staging \
    -c "SELECT archived_count, failed_count, last_archived_wal, last_archived_time FROM pg_stat_archiver;"'
```

### Alarm thresholds

| Condition | Severity | Action |
|-----------|----------|--------|
| `failed_count > 0` | ⚠️ warning | См. [Troubleshooting § failed_count растёт](#failed_count-растёт) |
| `spool > 10 файлов` | 🔴 critical | См. [Troubleshooting § Spool overflow](#spool-overflow) |
| `archived_count` не растёт 2+ часа | 🔴 critical | postgres archive_command broken или spool push stuck |
| R2 bucket empty для >2 часов | 🔴 critical | Полный flow broken — investigate |

### Weekly full backup

**Автомат:** cron `/etc/cron.d/proficrm-walg-retention` в воскресенье 02:00 UTC.

**Ручной запуск (ad-hoc):**

```bash
ssh -i ~/.ssh/id_proficrm_deploy root@5.181.254.172 \
  'cd /opt/proficrm-staging && \
   docker compose -f docker-compose.walg-backup.yml run --rm walg-backup'
```

**Ожидаемо:** ~1-2 мин duration, ~1 GB compressed output, exit 0. Помни:
при первом запуске на не-использованном VPS `docker compose` может pull
`postgres:16` image (~150 МБ) дополнительно.

## Emergency PITR restore

### Step 1: Подготовить drill environment

```bash
ssh -i ~/.ssh/id_proficrm_deploy root@5.181.254.172 bash <<'SSH'
set -e
# Restore drill compose (удалён после 3.2 cleanup, в репо есть).
cd /opt/proficrm-staging
git show HEAD:docker-compose.walg-drill.yml > docker-compose.walg-drill.yml

# Prepare data dir.
rm -rf /tmp/walg-drill-data
mkdir -p /tmp/walg-drill-data
chown 999:999 /tmp/walg-drill-data
chmod 700 /tmp/walg-drill-data

# Start drill container (sleeps).
docker compose -f docker-compose.walg-drill.yml up -d
sleep 5
docker compose -f docker-compose.walg-drill.yml ps
SSH
```

### Step 2: backup-fetch LATEST

```bash
ssh -i ~/.ssh/id_proficrm_deploy root@5.181.254.172 \
  'cd /opt/proficrm-staging && \
   docker compose -f docker-compose.walg-drill.yml exec -T db-drill bash -c "
     set -a; . /etc/wal-g/walg.env; set +a
     chown -R postgres:postgres /var/lib/postgresql/data
     chmod 700 /var/lib/postgresql/data
     su postgres -c \"/usr/local/bin/wal-g backup-fetch /var/lib/postgresql/data LATEST\"
   "'
```

**Ожидаемо:** ~10-15 секунд для 1 GB compressed base backup.

### Step 3: Setup recovery.signal + restore_command

```bash
# Create wrapper on host first (избегаем quote escaping боль).
ssh -i ~/.ssh/id_proficrm_deploy root@5.181.254.172 "cat > /tmp/restore-command-host.sh" <<'EOF'
#!/bin/bash
set -a
. /etc/wal-g/walg.env
set +a
exec /usr/local/bin/wal-g wal-fetch "$1" "$2"
EOF

# Copy to drill + chmod + chown.
ssh -i ~/.ssh/id_proficrm_deploy root@5.181.254.172 \
  'docker cp /tmp/restore-command-host.sh proficrm-staging-db-drill-1:/tmp/restore-command.sh && \
   docker exec proficrm-staging-db-drill-1 chmod +x /tmp/restore-command.sh && \
   docker exec proficrm-staging-db-drill-1 chown postgres:postgres /tmp/restore-command.sh'

# Write postgresql.auto.conf (archive off, restore_command set).
ssh -i ~/.ssh/id_proficrm_deploy root@5.181.254.172 \
  'docker compose -f /opt/proficrm-staging/docker-compose.walg-drill.yml exec -T db-drill bash -c "
    cat > /var/lib/postgresql/data/postgresql.auto.conf <<CONF
archive_mode = '"'"'off'"'"'
archive_command = '"'"''"'"'
archive_timeout = '"'"'0'"'"'
wal_level = '"'"'replica'"'"'
restore_command = '"'"'/tmp/restore-command.sh %f %p'"'"'
CONF
    chown postgres:postgres /var/lib/postgresql/data/postgresql.auto.conf
    touch /var/lib/postgresql/data/recovery.signal
    chown postgres:postgres /var/lib/postgresql/data/recovery.signal
    : > /var/lib/postgresql/data/drill.log
    chown postgres:postgres /var/lib/postgresql/data/drill.log
  "'
```

**PITR к конкретной точке:** добавить `recovery_target_time = '<ISO8601>'`
в `postgresql.auto.conf` перед `recovery.signal`. См. PostgreSQL docs,
[PITR reference](https://www.postgresql.org/docs/16/continuous-archiving.html).

### Step 4: Start postgres, wait recovery done

```bash
ssh -i ~/.ssh/id_proficrm_deploy root@5.181.254.172 \
  'docker compose -f /opt/proficrm-staging/docker-compose.walg-drill.yml exec -T db-drill \
    su postgres -c "pg_ctl -D /var/lib/postgresql/data -l /var/lib/postgresql/data/drill.log -o \"-p 5433\" -t 120 start"'

# Verify recovery complete.
ssh -i ~/.ssh/id_proficrm_deploy root@5.181.254.172 \
  'docker compose -f /opt/proficrm-staging/docker-compose.walg-drill.yml exec -T db-drill \
    su postgres -c "psql -p 5433 -U crm_staging -d crm_staging -tAc \"SELECT pg_is_in_recovery();\""'
# Expected output: f
```

### Step 5: Validation queries

```bash
# Primary (staging на 15432).
ssh -i ~/.ssh/id_proficrm_deploy root@5.181.254.172 \
  'docker compose -f /opt/proficrm-staging/docker-compose.staging.yml -p proficrm-staging exec -T db \
    psql -U crm_staging -d crm_staging \
    -c "SELECT COUNT(*) FROM accounts_user, (SELECT COUNT(*) FROM companies_company) c, (SELECT COUNT(*) FROM audit_activityevent) a;"'

# Drill (на 5433) — должны совпадать до последних нескольких минут transactions.
ssh -i ~/.ssh/id_proficrm_deploy root@5.181.254.172 \
  'docker compose -f /opt/proficrm-staging/docker-compose.walg-drill.yml exec -T db-drill \
    su postgres -c "psql -p 5433 -U crm_staging -d crm_staging \
      -c \"SELECT COUNT(*) FROM accounts_user;\""'
```

**Success criterion:** row counts совпадают с точностью до recent transactions.

### Step 6: Cleanup после drill

```bash
ssh -i ~/.ssh/id_proficrm_deploy root@5.181.254.172 \
  'cd /opt/proficrm-staging && \
   docker compose -f docker-compose.walg-drill.yml down -v && \
   rm -rf /tmp/walg-drill-data && \
   rm docker-compose.walg-drill.yml'
```

## Fallback — host-direct backup-push

Если helper compose сломается (например, новая Debian 12 image drops CA bundle
вообще), можно запустить `wal-g backup-push` прямо с хоста:

**Требование:** symlink `/var/lib/postgresql/data` → docker volume.

```bash
ssh -i ~/.ssh/id_proficrm_deploy root@5.181.254.172 \
  'mkdir -p /var/lib/postgresql && \
   ln -sfn /var/lib/docker/volumes/proficrm-staging_pgdata_staging/_data /var/lib/postgresql/data'
```

Восстановить script из репо (commit `a13bf9a6`):

```bash
ssh -i ~/.ssh/id_proficrm_deploy root@5.181.254.172 \
  'cd /opt/proficrm-staging && git show HEAD:scripts/walg-backup-host.sh > /tmp/walg-backup-host.sh && chmod +x /tmp/walg-backup-host.sh'
```

Запуск:

```bash
ssh -i ~/.ssh/id_proficrm_deploy root@5.181.254.172 \
  'setsid nohup /tmp/walg-backup-host.sh </dev/null >/tmp/walg-backup-host.log 2>&1 & disown'
```

Monitor:

```bash
ssh -i ~/.ssh/id_proficrm_deploy root@5.181.254.172 \
  'until grep -q "\[host-backup\] exit=" /tmp/walg-backup-host.log; do sleep 15; done; tail -20 /tmp/walg-backup-host.log'
```

## Troubleshooting

### `x509: certificate signed by unknown authority` в container

**Симптом:** wal-g в postgres:16 container падает на HTTPS к R2.

**Причина:** Debian 12 CA bundle не trust'ит Cloudflare chain.

**Fix:** bind mount `/etc/ssl/certs:/etc/ssl/certs:ro` в helper compose.
Уже применено в `docker-compose.walg-backup.yml` (commit `a13bf9a6`).

Если видишь эту ошибку в новом helper — добавь volume:

```yaml
volumes:
  - /etc/ssl/certs:/etc/ssl/certs:ro
  # ... остальные mounts
```

### failed_count растёт

```bash
# 1. Смотрим cron log.
tail -50 /var/log/proficrm-walg-spool.log

# 2. Verify wal-g can push из host-level.
ssh root@5.181.254.172 \
  'set -a; . /etc/wal-g/walg.env; set +a; /usr/local/bin/wal-g st ls wal_005/ | tail -5'

# 3. Test manual push.
ssh root@5.181.254.172 \
  'ls /var/lib/proficrm-staging/wal-spool/ | head -1 | while read f; do \
     set -a; . /etc/wal-g/walg.env; set +a; \
     /usr/local/bin/wal-g wal-push /var/lib/proficrm-staging/wal-spool/"$f"; \
   done'
```

### Spool overflow

**Симптом:** `ls /var/lib/proficrm-staging/wal-spool/` возвращает много файлов.

**Причины и fix:**

```bash
# 1. Cron жив?
ssh root@5.181.254.172 'systemctl is-active cron && systemctl status cron | head -10'

# 2. Cron config читается?
ssh root@5.181.254.172 'ls -la /etc/cron.d/proficrm-walg-spool && cat /etc/cron.d/proficrm-walg-spool'

# 3. Последний tick cron push script?
ssh root@5.181.254.172 'tail -30 /var/log/proficrm-walg-spool.log'

# 4. Lockfile stuck? (не должно, но check).
ssh root@5.181.254.172 'ls -la /var/run/proficrm-walg-push.lock'
```

**Bruteforce manual drain:**

```bash
ssh root@5.181.254.172 '/opt/proficrm-staging/scripts/walg-push-from-spool.sh'
```

### R2 auth errors

**Симптом:** `AccessDenied` / `InvalidAccessKeyId` в логах wal-g.

**Fix:**

1. Проверить `/etc/wal-g/walg.env` на наличие и non-empty `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`.
2. Cloudflare dashboard — R2 API Tokens, сроки истечения (по умолчанию permanent, но проверь).
3. Regenerate token если expired, update walg.env, chmod 600, test `wal-g st ls wal_005/`.

## Migration к MinIO (future W10.1 proper)

**Триггер:** future W10.1 proper session deploys self-hosted MinIO (см.
`docs/audit/hotlist.md` item «W10.1 proper MinIO setup»).

**При migration revisit:**

- **In-container wal-g** может стать жизнеспособным опять — MinIO internal
  endpoint (`http://minio:9000`) не требует внешнего HTTPS CA trust. Host-pivot
  может быть упразднён в пользу прямого `archive_command = 'wal-g wal-push %p'`
  в container, что проще операционно.
- **Или оставить host-pivot** если CA bundle issue shows up снова.

См. `docs/decisions/2026-04-24-wal-g-r2-bridge-to-minio.md` §Migration plan.
